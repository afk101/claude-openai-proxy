# Chat Completions 自动断连排查记录

## 用户报告

- 服务在使用过程中会自动断开。
- 需要基于 `.env` 中的密钥和上游地址，通过 `curl` 直接验证。
- 验证范围包含 OpenAI Chat Completions 的常见请求格式、流式调用与工具调用。
- 客户端允许的 `model` 仅为 `Auto`、`360-deepseek-v4-flash`、`360-glm-5.2`，三个模型的上下文窗口均为 200k。
- 向 Claude Responses 兼容上游转换时，模型字段必须原样透传，不能映射为 Claude 模型名。

## 初步代码与历史检查

- 最近提交 `4680028` 修改了 `max_tokens` 动态分配；此前提交引入了 Claude 模型名映射。该历史与当前“模型必须原样透传”的要求存在潜在冲突，需逐行检查转换逻辑。
- 当前流式路径在 `_stream_with_disconnect_check` 中仅在产出上游转换后的 chunk 后检查客户端是否断开；上游 `create_message_stream` 以 `aiter_lines()` 读取，并会忽略空行。
- 当前非流式客户端在 `create_message` 中创建后台任务和取消等待任务；需通过真实请求和服务日志确认断连发生在哪个边界。
- `.env` 已配置上游密钥和上游地址；未在终端输出敏感值。
- 当前上游地址为 HTTPS 服务，代理端口为 `7072`，且上游密钥已配置；代理未配置客户端访问密钥。
- 当前转换层已将收到的 `model` 原样写入上游请求，但没有白名单校验。`resolve_max_tokens` 仍按旧 Claude 模型名映射，未知模型会回退为 `4096`，与三个指定模型均为 200k 上下文的要求不符。
- 当前测试同样使用旧 Claude/GPT 模型名，无法证明新的模型约束与透传要求。

## 当前结论

- 新增模型契约测试已在未修改生产代码时运行：`Auto` 默认被解析为 `4096`（期望 `128000`）；白名单外 `claude-4.8-opus` 被接口接受（期望 `422`）。这确认了模型约束问题。
- 进一步代码追踪发现：上游流式客户端以 `aiter_lines()` 逐行产出且丢弃空行，而转换器需要同一字符串中同时存在 `data: ...` 才能解析。真实 Anthropic SSE 的事件字段与数据字段通常被空行分隔为一个事件；现有单元测试将整段事件一次性 yield，未覆盖此组件边界。该路径是“流式自动断开/无内容完成”的主要待验证假设。
- 尚未修改生产代码；下一步先添加脱敏诊断日志，以真实 curl 的上游事件记录确认假设。

## 已完成的诊断准备

- 将允许模型及其 200k 值集中至 `src/core/constants.py`，并由请求模型使用该常量进行校验；转换函数继续原样透传模型。
- 针对这三项契约和日志的定向测试已通过：`4 passed`。完整真实链路验证尚未执行。
- 首次真实 curl 未进入代理：后台子进程在启动命令会话结束后退出，后续 curl 返回连接失败（HTTP `000`）。健康检查曾成功，证明配置可加载；这属于测试进程生命周期问题，不是代理业务断连。后续将使用保持常驻的终端会话，避免重复该启动方式。

## 真实请求证据

- 常驻服务上的非流式 curl（模型 `Auto`、`max_tokens=32`）返回 HTTP `200`，响应为 `chat.completion`，模型仍为 `Auto`，正文为“连通”。代理日志确认上游 POST 也返回 HTTP `200`。
- 常驻服务上的流式 curl（模型 `360-deepseek-v4-flash`、`max_tokens=32`）返回 HTTP `200`，具备 `[DONE]`，且输出模型仍为原始模型。上游的 `finish_reason` 为 `length`，仅输出一个字符“流”。
- 此流式结果证明逐行读取并未导致事件完全丢失：`data:` 行可独立被转换器解析。因此“事件字段和数据字段被拆开”不是本次实际断开根因，已排除。
- 当前待验证假设：显式较小的 `max_tokens` 可能被上游模型内部推理/输出预算迅速消耗；代理尚未记录上游实际提交的 `max_tokens`，下一步会补充脱敏字段日志并以省略 token 上限的请求验证 200k 默认值。

## 默认上限验证

- 新增上游请求日志（模型、流式标志、`max_tokens`）的测试已先失败后通过；服务重启后，省略 `max_tokens` 的流式 curl 使用模型 `360-glm-5.2` 返回 HTTP `200`。
- 该请求日志明确记录 `model=360-glm-5.2`、`max_tokens=128000`；SSE 完整输出“默认上限连通”，以 `stop` 和 `[DONE]` 正常结束。
- 因此小上限请求的单字符 `length` 是上游输出预算耗尽，不是连接断开。模型透传和 200k 默认值已在真实上游验证。

## 工具调用首次验证

- 真实非流式工具请求（模型 `Auto`，OpenAI `tools` + 指定函数 `tool_choice`）返回 HTTP `200`，上游确实调用了 `get_weather`，参数为北京；输出模型仍为 `Auto`。
- 发现协议错误：响应含有有效 `tool_calls`，但 `finish_reason` 为 `stop`。这是上游以 `end_turn` 结束工具调用时，当前转换器机械映射 stop reason 导致的；标准 OpenAI 客户端应收到 `tool_calls`。将用回归测试修复后，再验证携带 `tool` 消息的下一回合。

## 工具结束原因修复

- 已新增回归测试，确认“上游 `end_turn` + `tool_use`”此前错误映射为 `stop`。
- 修复后转换器优先根据已经生成的 OpenAI `tool_calls` 判定结束原因；该回归测试和原本的 `tool_use` 测试均通过（`2 passed`）。
- 测试迁移过程中发现一条仍期望 `gpt-4o` 的旧断言，已改为允许模型 `Auto`；这不改变生产逻辑。

## 工具结果与流式工具调用验证

- 修复后的真实两回合工具调用使用 `360-deepseek-v4-flash`：首次返回 `finish_reason=tool_calls`，随后包含 `assistant.tool_calls` 与 `tool` 结果的请求返回 HTTP `200`、自然语言结果及 `stop`。
- 真实流式工具调用（`Auto`）产生 17 个工具增量，含工具 ID/名称，并以 `[DONE]` 结束，但最后错误给出 `finish_reason=stop`。
- 根因与非流式分支相同：流式 `message_delta` 仅映射上游停止原因，未检查流状态中已记录的工具调用。下一步将以测试先行修复。
